# AGENTS.md

このファイルは、このリポジトリで作業するAI Agent向けの実務ルールです。  
一般的な開発ルールに加えて、**このプロジェクト固有の制約**を明記します。

## 1. 基本方針（一般）

- 変更は最小単位で行い、無関係なファイルを触らない。
- まず既存実装に合わせる（命名・構造・責務分割を壊さない）。
- 推測で仕様を変えない。曖昧な場合は、実装から読み取れる挙動を維持する。
- 可能な限り型安全に実装し、`any` を増やさない。
- 秘匿情報（トークン、個人情報、絶対パス等）をコード・ログ・ドキュメントに残さない。

## 2. 技術スタックと責務（プロジェクト固有）

- Frontend: React + TypeScript + Vite
- Backend: Vercel Functions（`api/`）
- DB: Neon(PostgreSQL)（`@neondatabase/serverless`）
- ローカル保存: Dexie(IndexedDB)

主な責務:

- `src/db.ts`: データアクセスのFacade（クラウド/ローカル切替）
- `src/cloudApi.ts`: `/api/*` 呼び出し
- `api/*.ts`: サーバーAPI（認証・所有権チェック含む）
- `src/types/index.ts`: 共有型

## 3. 実装ルール（プロジェクト固有）

### 3.1 データアクセス

- データ追加/更新系の実装は、`src/db.ts` のクラウド・ローカル両方の分岐整合を保つ。
- Cloud Sync判定は `localStorage["useCloudSync"] === "true"` を前提に既存設計へ合わせる。

### 3.2 API

- APIハンドラは `api/_http.ts` の `ApiHandlerRequest/ApiHandlerResponse` を使う。
- データ系APIでは、認証チェック + ユーザー所有権チェックを必須にする。
- 500系レスポンスで内部例外メッセージ（`err.message`）をクライアントへ返さない。
- 認証情報・パスワード・個人情報を `console.log` しない。

### 3.3 DB/マイグレーション

- スキーマ変更は `api/init.ts` に追記し、`IF NOT EXISTS` を使って冪等にする。
- 既存データがある前提で、制約追加前に必要なクリーンアップを行う。

### 3.4 日付・復習ロジック

- 復習日 (`nextDue`) は `YYYY-MM-DD` のローカル日付基準で扱う。
- `toISOString().slice(0, 10)` で日付計算しない（UTCずれ防止）。

### 3.5 UI共通化

- 問題集未発見表示は `src/components/NotFoundView.tsx` を優先利用。
- Study/Memorization系の共通フレームは `src/components/QuizSessionLayout.tsx` を優先利用。
- ルートの `:id` から `activeQuizSet` 解決は `src/hooks/useActiveQuizSetFromRoute.ts` を優先利用。

## 4. 品質チェック

変更後は最低限以下を実行:

- `npx eslint <変更したファイル...>`
- `npm run build`

大きい変更時は必要に応じて:

- `npm run lint`

## 5. 変更時の注意

- `dist/` はビルド成果物。ソース修正対象ではない。
- UI文言・ヘルプ文言は既存トーン（日本語中心）に合わせる。
- 既存ユーザーのデータ互換性を壊す変更（型変更、JSON構造変更）は慎重に扱う。

## 6. バージョン管理・リリースノート（必須）

このプロジェクトでは、**タスク完了時（最終報告直前）**に以下を必ず実施する。

### 6.1 `package.json` の `version` 更新

- 対象: `package.json`
- ルール:
  - **Major**: 破壊的変更、全体再設計、既存利用者に移行作業が必要
  - **Minor**: 互換性を維持した新機能追加、新画面追加
  - **Patch**: バグ修正、UI微調整、リファクタリング、内部改善
- 複数変更を同時に含む場合は、最も影響の大きいレベルを採用する。

### 6.2 リリースノート追記

- 対象: `src/pages/ReleaseNotesRoute.tsx`
- 追記位置: 既存の `release-card` 一覧の**最上部**。
- 形式:
  - `h2` に `vX.Y.Z`
  - `span` に当日の日付（既存表記に合わせて `YYYY年M月D日 HH:MM`）
  - 見出しは以下を必ず含める:
    - `✨ 新機能・改善`
    - `🐛 バグ修正`
- 記載内容は実装詳細ではなく、**ユーザーにとって何が変わるか**を日本語で簡潔に書く。

### 6.3 実務上の運用ルール（推奨）

- 同一タスク中にコミットが複数ある場合、バージョン更新とリリースノート追記は最終コミットでまとめて実施してよい。
- ユーザー影響がない変更で項目がない場合でも、見出しは維持し、`変更なし` と明記する。

## 7. Codex運用フロー（Boris風）

このプロジェクトでは、Codex での作業時に以下の流れを基本とする。

### 7.1 基本フロー
1. **Understand**
   - 依頼内容を短く整理し、影響範囲を特定する。
   - まず既存実装・既存UI・既存APIを確認し、推測で新責務を作らない。

2. **Plan**
   - 変更対象ファイルを明示する。
   - 変更しないファイル・責務も意識する。
   - 既存コンポーネント / hook / API / 型の再利用可否を先に確認する。

3. **Execute**
   - 変更は最小単位で行う。
   - UI変更は既存トーン（日本語中心）と既存レイアウトを優先する。
   - API変更は認証・所有権チェック・エラーハンドリングを必ず確認する。
   - クラウド/ローカル両対応が必要な箇所では `src/db.ts` の分岐整合を壊さない。

4. **Verify**
   - 変更後は最低限、変更ファイルに対する ESLint と build を通す。
   - 必要に応じて関連画面の導線・状態遷移・回帰影響を確認する。
   - 日付処理を変更した場合は UTC ずれがないか確認する。

5. **Close**
   - ユーザー影響がある場合は `package.json` の version を更新する。
   - `src/pages/ReleaseNotesRoute.tsx` にリリースノートを追記する。
   - 最終報告では「何を変えたか」「影響範囲」「確認結果」を簡潔にまとめる。

### 7.2 Codexが優先して確認すべきファイル
- `AGENTS.md`
- `docs/ai/project-context.md`
- `docs/ai/conventions.md`
- `docs/ai/decision-log.md`
- `docs/ai/progress.md`
- 変更対象配下の `AGENTS.md`（例: `src/AGENTS.md`, `api/AGENTS.md`）

### 7.3 Codex向けの追加ルール
- 「新しい仕組みを足す」前に、既存の hook / component / utility で吸収できないか確認する。
- 仕様が曖昧な場合、実装から読み取れる既存挙動を優先して維持する。
- 一時的なデバッグコード、不要な `console.log`、説明不足の `any` を残さない。
- サーバー例外の詳細をクライアントに返さない。
- セキュリティ・認可・データ互換性に関わる変更は、小さく分けて扱う。

### 7.4 Boris風の役割分担の読み替え
このリポジトリでは専用エージェント群を持たないため、Codex は以下の観点を自分で順に満たす。

- **Architect**: 既存責務を壊していないか確認する
- **Implementer**: 最小差分で実装する
- **Reviewer**: 変更の過不足・命名・責務分割を見直す
- **Verifier**: lint / build / 必要な動作確認を行う
- **Documenter**: 必要に応じて release notes / docs を更新する

### 7.5 変更前チェックリスト
- どの責務層の変更か明確か
- 既存の共通部品を再利用できないか確認したか
- Cloud / Local 両方に影響する変更か把握したか
- ユーザー影響の有無を把握したか

### 7.6 変更後チェックリスト
- 変更対象ファイルに lint をかけたか
- build が通るか
- 認証 / 所有権 / 日付 / 型安全性を壊していないか
- ユーザー影響がある場合、version / release notes を更新したか
