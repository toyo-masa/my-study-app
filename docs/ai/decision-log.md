# Decision Log

## 2026-04-12
### 住宅ローン画面の物件リンクは下部の独立カードとして localStorage 保存する
- 物件リンクは保存済み条件や返済入力とは混ぜず、ページ下部の専用カードに分ける
- 保存内容は `物件名/メモ + URL + 更新時刻` とし、front-end only の方針に合わせて localStorage に閉じる
- URL は http/https のみ受け付け、一覧からそのまま開けるようにする

### 理由
- 物件ページの参照は返済条件そのものとは別の文脈なので、下部の補助情報として分けた方が責務が明確
- Notion 埋め込み前提でも、主要な試算 UI を圧迫せずにメモ欄を足しやすい
- まずはシンプルな保存場所があれば用途を満たせるため、認証や DB を増やさずローカル保存で十分

### 住宅ローン条件の保存参照は一覧よりプルダウン即時反映を優先する
- 保存済み条件は一覧から選んで読み込む形ではなく、ネイティブ `select` で選んだ瞬間に入力欄へ反映する
- 一覧管理は通常表示から外し、削除だけを行う折りたたみ管理へ退避する
- 保存名入力は残し、同名保存で上書きする従来ルールは維持する

### 理由
- 比較用途では「候補を切り替えてすぐ結果を見る」方が主で、一覧から毎回ボタンを押す導線は重い
- ネイティブ `select` の方が既存依存を増やさず、モバイルや埋め込みでも安定しやすい
- 削除操作は必要だが、常時一覧を見せるとフォームの高さを押し上げるため、通常時は隠した方が見通しがよい

### 住宅ローン画面はサマリを上段全面、入力条件を下段2カラムへ再配置する
- `入力条件左 / サマリ右` の構成はやめ、サマリを最上段の全面表示へ移す
- 入力条件カード内では `借入条件` と `積立・固定費` を desktop で 2 カラムにし、狭い幅では 1 カラムへ落とす
- サマリは主要指標と補助指標の 2 段構成に分け、比較で見たい値を先に集める

### 理由
- 入力フォームが長い一方でサマリが短い現状では、左右に並べると高さ差が大きく見えて視線移動が増える
- サマリを先に見せると、条件変更後に確認したい値へ視線を戻しやすい
- 入力条件自体は項目数が多いため、縦積みより 2 カラムの方が desktop 幅を使いやすい

### 住宅ローンの推移グラフは既存 SVG のまま最寄り月ホバーで詳細を出す
- `loan-sim` のグラフはライブラリ差し替えをせず、既存の自前 SVG 描画を維持する
- 詳細表示は各点に大量の DOM を置かず、マウス位置から最寄りの月インデックスを逆算して tooltip を出す
- 表示内容はまず年月と対象系列の金額に絞り、チャートの見た目や高さは大きく変えない

### 理由
- 返済期間が長いと点数が多くなり、点ごとの hover 要素を増やすと描画と管理が重くなりやすい
- 既存の `loan-sim` グラフは SVG で完結しているため、局所的な hover 追加の方が責務と依存を増やさずに済む
- Notion 埋め込み前提でも、チャートカードの高さを増やさずに詳細を見られる方が扱いやすい

### 住宅ローン返済シミュレーターの条件保存は localStorage の名前付きプリセットで扱う
- 保存先はサーバーや DB を増やさず、`loan-sim` 専用の `localStorage` に閉じる
- 保存形式は「名前・入力条件スナップショット・更新時刻」を持つ配列とし、同名保存は上書きにする
- UI はフォームカード内で完結させ、別モーダルや別ページは増やさない

### 理由
- このツールは front-end only の単独アプリとして追加しており、比較条件の保持もクライアント内に閉じる方が既存方針に合う
- 名前付き保存にしておくと、複数の借入パターンを比較したい用途にそのまま使いやすい
- モーダルや別画面を増やさない方が、Notion 埋め込み前提の縦長レイアウトを崩しにくい

### 年収に対する返済負担は「税込年収」と「概算手取り」の両方で示す
- 住宅ローン返済シミュレーターに年収入力を追加し、毎月返済額が月換算年収と概算手取りの何割かをサマリーへ出す
- 手取りは、独身会社員・40歳未満・東京都の協会けんぽ・一般事業・扶養や賞与なし前提で概算する
- 概算は UI 上でも厳密額ではなく比較用の目安として扱い、返済計算本体とは分離した helper で閉じる

### 理由
- 返済額の絶対値だけでは、収入に対して重いかどうかを直感的に判断しづらい
- 一方で、家族構成や控除をすべて入力させると画面が重くなるため、まずは比較用の単純な目安に留める方が既存 UI と相性がよい
- 概算ロジックを返済ロジック本体から切り離しておくと、将来前提条件を見直すときも影響範囲を限定しやすい

### 住宅ローン返済シミュレーターは front-end only の単独アプリとして追加する
- `/loan-sim` は既存の単独学習アプリと同じく、AppLauncher -> Route ラッパ -> 専用コンポーネントの流れで追加する
- API / DB / 認証 / ローカル保存は増やさず、入力変更時のクライアント再計算だけで完結させる
- 計算ロジックは `src/features/loanSim` に純関数として分離し、UI とは切り離す

### 理由
- 既存アプリ全体の責務を壊さず、追加機能を最小差分で差し込みやすい
- 将来 Notion 埋め込みへ持っていく前段として、フロント単体で検証しやすい
- 計算ロジックを分離しておくと、返済条件や表示 UI を後で変えても検証しやすい

### 住宅ローン計算ではボーナス返済と積立ルールを固定して明示する
- ボーナス返済は、開始から 6 か月ごとの月末に「元本への追加返済」として扱う
- 積立は、前月残高に月利を反映してから月末積立を加える順序で扱う
- 返済表は返済年数ぶんの全期間を持ち、ローン完済後も固定費と積立は継続して表示する

### 理由
- ユーザーが比較したいのは、返済そのものだけでなく、積立込みの月次キャッシュフローと完済後の残高だから
- ルールを固定しないと、ボーナス返済や積立残高の見え方がケースごとにぶれやすい
- 完済後も同じ期間で並べておくと、「ローン完済時点の積立残高」と「計画終了時の積立最終額」を分けて見やすい

## 2026-04-11
### ホーム画面の問題集カードはタグフィルタと並び替えを同じ行で分ける
- ホーム画面の問題集一覧は、左側にタグフィルタ、右側に並び替え UI を置く
- 並び替え軸は作成順・問題数順・あいうえお順の3つに限定し、昇順 / 降順は別トグルで切り替える
- 並び替えは表示用配列だけに適用し、`quizSets` 自体の保持順や既存の追加・復元ロジックは変えない

### 理由
- フィルタとソートを同じ帯の左右に置いた方が、一覧操作の導線が散らばらず認識しやすい
- 軸選択と昇降順を分けると、既存 UI を大きく崩さずに必要な組み合わせを増やせる
- 元データを並び替えない構成なら、ホーム画面以外の既存フローへの影響を最小に抑えやすい

### 正解時の復習日分散は `nextDue` 確定時に一度だけ offset を決める
- 復習日の後ろ倒しは `intervalDays` や `consecutiveCorrect` に混ぜず、保存する `nextDue` を決めるタイミングでだけ `offsetDays` を解決する
- `offsetDays` 決定 helper と日付確定 helper を分離し、同一更新処理の中では一度決めた `nextDue` を schedule / log の両方で共有する
- 分散は正解時だけを対象とし、不正解・自信なし時は従来どおり固定間隔のまま扱う

### 理由
- helper 内で毎回乱数を引くと、再計算や二重送信時に `nextDue` がぶれる危険がある
- 分散の目的は復習日の一点集中を軽く緩和することであり、学習難易度や定着度の推定まで変えない方が既存ロジックを壊しにくい
- `nextDue` だけを後ろへずらす構成なら、Cloud / Local の既存データ構造を変えずに安全に導入しやすい

### 統計分布表の初期サイズは電卓と同じ基準値へそろえる
- 統計分布表パネルの初期 width / height は専用の大きめ値を持たず、電卓の基準サイズ定数をそのまま再利用する
- 分布表側の最小サイズ clamp は維持し、初期表示だけをそろえる

### 理由
- 同じ右上ツールから開く補助パネルとして、最初の占有面積が大きく異なると操作感がばらつく
- 初期値の統一は定数参照の差し替えだけで済み、既存のリサイズ挙動や最小サイズ制約を壊さずに調整できる

### ツールパネルの閉じるボタン判定は HTMLElement ではなく Element 基準にする
- 電卓と統計分布表のヘッダードラッグ開始時、`event.target` が `svg` / `path` でも `closest('button')` を判定できるよう `Element` 基準で分岐する
- ドラッグヘッダー上の閉じるボタンは、アイコン本体を押した場合でも drag 開始を抑止する

### 理由
- `X` アイコン直上では `event.target` が `HTMLElement` ではなく SVG 系要素になることがあり、既存条件だと「ボタン上ではない」と誤判定される
- 電卓と統計分布表で同じドラッグヘッダー構造を使っているため、判定条件をそろえて直すのが最小差分で確実

### t分布表の自由度列は共通幅のままにせず個別に詰める
- 統計分布表の先頭列スタイルは共通の sticky 列を維持しつつ、`t` テーブルだけ modifier class で自由度列幅を狭める
- 正規分布表やカイ二乗分布表の先頭列幅には手を入れず、横幅の最適化が必要な表だけ局所調整する

### 理由
- t 分布表は自由度ラベルが短く、共通幅だと値列の横幅を不必要に消費しやすい
- 表ごとに列構成が異なるため、共通 CSS 全体を縮めるより modifier で分けた方が回帰を抑えやすい

### 問題回答画面の統計分布表も電卓と同じヘッダードラッグで移動できるようにする
- 統計分布表ツールは中央固定のままにせず、ヘッダー部分の pointer drag で `left` / `top` を更新する
- リサイズ時も現在位置を基準に viewport 内へ収まるように clamp し、電卓と同じ「移動後の位置を保ったままサイズ変更できる」挙動へそろえる

### 理由
- 問題画面では問題文・選択肢・手書きメモと重なりやすく、中央固定だと参照位置を逃がせない
- 既存の電卓ドラッグ実装に寄せた方が、学習中ツールの操作感をそろえつつ差分を最小にできる

## 2026-04-10
### 統計分布表パネルのリサイズは縮尺変更ではなくコンテナサイズ変更で扱う
- 問題画面の統計分布表は、パネル右下のハンドルでコンテナサイズだけを調整し、表本体は内部スクロールで読む
- 電卓のような中身全体のスケーリングは行わず、表の文字サイズやセル密度は維持する

### 理由
- 分布表は表の可読性が優先で、縮尺まで落とすとセル値を読み取りにくくなる
- 既存の埋め込み `DistributionTables` は内部スクロール前提の構成なので、コンテナだけ可変にした方が最小差分で済む

### 電卓の `1/x` は新演算子ではなく既存式ラッパーとして扱う
- `1/x` ボタンは専用の評価構文を増やさず、末尾の値や式を `1÷(…)` で包む入力として実装する
- 空の状態や演算子直後では `1÷(` を挿入し、既存の括弧補完・パーサ・表示ロジックをそのまま使う

### 理由
- 既存の再帰下降パーサは `÷` と括弧で十分に逆数を表現でき、新しい関数トークンを増やす必要がない
- `sqrt` などの「直前の値を包む」操作と同じ流れで実装した方が、既存 UI の一貫性を保ちやすい

### 問題回答画面の統計分布表は既存コンポーネントを埋め込み再利用する
- `DistributionTables` にヘッダー表示・プレビュー表示の切り替えだけを足し、問題回答画面では表のみの埋め込みモードで使う
- 新しい分布表専用コンポーネントは増やさず、ルート画面とツールパネルで同じ表データ定義を共有する

### 理由
- 正規分布表・t 分布表・カイ二乗分布表の中身は既存の `DistributionTables` で完結しており、表だけ欲しい要件は表示切り替えで吸収できる
- データ定義やセル選択ロジックの二重管理を避け、問題画面用の追加差分を最小に保ちやすい

### 手書きメモ欄の自動スクロール先は window ではなく学習画面内スクロールへ寄せる
- 手書きメモ欄リサイズ中の自動スクロールは `window.scrollBy` 前提にせず、学習画面の `.content-area` を優先してスクロールする
- 学習ルートで app shell を fixed 化したあとでも、メモ欄の下方向拡張と iPad のパン抑止を両立できる構成を優先する

### 理由
- remote 側の自動スクロール改善と、local 側の app shell 固定化はそのままではスクロール対象が食い違う
- 実際に動いているスクロールコンテナへ揃えた方が、空白方向のパンを抑えつつリサイズも継続しやすい

### 手書きメモ欄のリサイズ中は root 全体を一時ロックする
- 手書きメモ欄の下端ドラッグ中だけ `html` / `body` に専用クラスを付け、`overflow` / `overscroll` / `touch-action` を止める
- リサイズ追従はハンドル要素内の `onPointerMove` ではなく、window 監視へ寄せて pointer capture と併用する

### 理由
- iPad 系では、リサイズハンドルを下へ引く操作がページ全体のパンや bounce と競合しやすい
- ドラッグ中だけブラウザ既定挙動を止める方が、通常スクロールを壊さずに症状へ直接効く

### 学習画面の app shell 自体を viewport に固定する
- `app-container.study-mode-active` を `position: fixed` + `inset: 0` にして、学習中の殻全体が viewport から外れないようにする
- ヘッダーは flex shrink を止め、戻るボタンやツール導線が常に上端に残る前提を強める

### 理由
- root のスクロールロックだけでは、iPad 系ブラウザで app shell 全体がずれたような状態に入ることがある
- 問題操作より優先して復帰導線を守る必要があり、ヘッダーを含む殻ごと固定した方が安全

### 学習画面では app shell だけでなく document root もスクロールロックする
- `study` / `memorization` ルートでは `app-container` だけでなく `html` / `body` にも専用クラスを付け、root 側の `overflow` と `overscroll` を止める
- 学習画面の通常スクロールは `content-area` に集約し、ブラウザ全体が空白方向へパンする余地を減らす

### 理由
- iPad 系ブラウザでは、内側コンテナを `overflow: hidden` にしていても document 側のパンが残ることがある
- 学習中は問題文・選択肢・手書きメモの操作が中心で、root スクロールを許可する利点より誤操作防止の優先度が高い

### 解答画面ヘッダーの右上操作は `headerActions` で拡張する
- `QuizSessionLayout` の右上領域は AI チャットボタン専用にせず、`headerActions` と既存 AI チャットトグルを並べられる構成にする

### 問題画面電卓の高度計算は独自パーサで扱う
- 問題画面の電卓は `eval` を使わず、括弧・冪乗・平方根・`log`・`ln`・`sin/cos/tan` を扱える再帰下降パーサで式を評価する
- 三角関数は度数法固定とし、`sin(90)=1` になる前提で扱う
- `x²` / `x³` は専用ボタン、一般の冪乗は `xʸ` と `^` で入力する
- 未閉鎖の括弧は表示上は薄い `)` で補助し、評価時は末尾で自動補完する
- バックスペースは文字単位ではなく、`sqrt(` や `sin(`、`^2` / `^3` のような入力単位でまとめて消す
- 式末尾が演算子などで未完成のまま `=` が押された場合はエラーにせず no-op にする
- ボタン配置は科学関数を上段、数字キーを下段、演算子を右列へ寄せた iPad 風レイアウトに寄せる
- 表示上の数値は 3 桁区切りで整形し、内部の評価式にはカンマを入れない
- リサイズ下限は内容量ベースで広めに取り、通常の viewport では全ボタンが枠内に収まる前提を優先する
- `±` は末尾の数値セグメントの符号だけを反転し、`%` は末尾数値を `÷100` に置き換える簡易入力として扱う
- 電卓ウィンドウのリサイズは縦横比固定に切り替え、縮小時は CSS 変数でボタンや文字サイズも比例して縮める
- `√` や `sin` などの前置関数は、直前の値があれば新規に `×` を挟まず、その値を包む形で入力する
- その後の操作性要望に合わせ、電卓のリサイズは再び縦横独立に戻し、縮小時の中身スケールだけを維持する
- さらに調整し、リサイズハンドルは右下のみに限定しつつ、ボタン群は残り高さいっぱいまで広がるレイアウトへ変更する
- 計算結果表示時は iPhone 電卓に寄せて、小さく薄い計算式を結果の上に残す
- 履歴表示行は条件付きで増減させず、最初から高さを確保して表示欄全体の揺れを防ぐ

### ベクトル・行列ラボは front-end only の単独 SVG アプリとして追加する
- ベクトル・行列ラボは API / DB / クラウド同期 / ローカル保存を増やさず、フロントエンドだけで完結する単独アプリとして追加する
- MVP は 2x2 行列と 2 次元平面に限定し、SVG ベースで単位正方形・単位円・基底・ベクトル・像/核を可視化する
- 可視化範囲は fixed 座標にせず auto-fit を前提にし、行列入力や合成結果で主要要素が見切れにくい構成を優先する
- AppLauncher -> Route ラッパ -> 単独コンポーネントの既存構成に合わせ、独立した `linear-algebra-lab-*` クラスで UI を持つ

### 理由（ベクトル・行列ラボ）
- 既存責務を壊さず、学習用アプリを最小差分で追加しやすい
- 外部依存や永続化を増やさず、挙動確認と保守をフロントエンド内で閉じられる
- 今後ほかの単独学習アプリを追加するときも、同じ導線と可視化方針を再利用しやすい

### ベクトル・行列ラボの非モバイル表示は「操作しながらグラフが見える」ことを優先する
- PC / iPad 幅では editor と visualization を縦積みにせず、2 列で同時表示する
- single plot のタブでは visualization を sticky に寄せ、入力カードを縦に追っても平面図を見失いにくくする
- summary は操作と可視化の邪魔をしない位置へ逃がし、まず変化を見られることを優先する
- グラフ中の `Ax` や `Ae1` などは、既存の KaTeX 依存を再利用して数式寄りに表示する
- タッチドラッグは pointer events と広めの hit area、`touch-action: none` でスクロール競合を抑える

### 理由（ベクトル・行列ラボ表示調整）
- この画面は「数値を変えた瞬間に平面図がどう変わるか」を見る用途が中心
- 縦積みレイアウトだと iPad 幅でも入力と可視化を行き来するスクロールが増える
- 数式ラベルの方が、線形代数の記法と図の対応を読み取りやすい
- タブレットやスマホの指操作では、小さい draggable handle だけだとスクロール競合が起きやすい

### ベクトル・行列ラボの記号表示は入力欄も KaTeX でそろえる
- グラフ中の `Ax` や `Ae1` だけでなく、入力欄の `a_{11}` `a_{12}` `x` `y` も既存の KaTeX 依存で描画する
- 数式記号の表示ルールをグラフと入力欄で分けず、同じ見た目で対応づけられることを優先する

### 理由（ベクトル・行列ラボ記号統一）
- 学習用途では、入力している成分と図中の記号が即座に対応して見える方が理解しやすい
- 既存依存で完結でき、追加ライブラリなしで UI の一貫性を上げられる

### iPad 系ブラウザでは SVG 内 `foreignObject` に依存しない
- ベクトル・行列ラボの graph ラベルは、SVG 内の `foreignObject` へ KaTeX HTML を埋め込まず、plot 上の HTML オーバーレイとして重ねる
- タッチ端末では sample vector のドラッグ開始点を handle だけに限定せず、plot 面からも開始できるようにする
- iPad 幅では editor を少し圧縮し、plot を相対的に大きく見せることを優先する

### 理由（iPad 向け graph 調整）
- iPad Safari 系では `foreignObject` の描画や配置が不安定になりやすい
- 学習用途では、ラベルが壊れないことと、指で直接動かしやすいことの方が厳密な埋め込み方式より重要
- touch 操作では、狭い handle だけを狙わせるより plot 面から触れた方が扱いやすい

### 理由
- 学習/暗記の両ルートへ同じ右上 UI を最小差分で差し込める
- 将来ツールを増やしても、AI チャット側の条件分岐を崩さず拡張しやすい
- 問題画面ショートカットとの干渉を、ツール UI ルート単位で管理しやすい
