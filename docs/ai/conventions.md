# Conventions

## 全体
- 変更は最小単位で行う
- 無関係なファイルを触らない
- 既存命名・既存構造・既存責務に寄せる
- 推測で仕様を変えない
- `any` を安易に増やさない
- 一時的なログやデバッグコードを残さない

## UI
- 既存トーンは日本語中心
- まず既存 component / hook の再利用を検討する
- 新規UIは既存画面から大きく浮かないようにする
- 共通レイアウトや NotFound 表示は既存共通部品を優先する

## データアクセス
- `src/db.ts` ではクラウド/ローカル両分岐の整合を保つ
- 片側だけ直してもう片側を壊す変更を避ける
- データ互換性を壊す JSON 構造変更・型変更は慎重に扱う

## API
- `api/_http.ts` の型を使う
- 認証チェックを必須にする
- 所有権チェックを必須にする
- 内部例外メッセージをそのまま返さない
- 認証情報・個人情報をログに出さない

## DB
- スキーマ変更は `api/init.ts` に追記する
- `IF NOT EXISTS` を使い冪等にする
- 制約追加前に既存データの整合性を考える

## 日付
- ローカル日付基準を守る
- UTC 変換前提の実装を避ける
- 復習日関連は特に既存挙動との差分に注意する

## 検証
- 変更ファイルに対する lint を優先
- build を必ず確認
- 必要なら関連画面の導線・表示崩れ・状態遷移も確認する
